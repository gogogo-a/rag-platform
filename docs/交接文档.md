## 数据流

文档上传：

1. API 保存文件和 MongoDB 文档记录。
2. API 提交后台处理任务并立即返回处理中。
3. `document_processor` 解析文本并分块。
4. `embedding_service` 生成向量。
5. `qdrant_client.upsert_documents()` 写入 Qdrant。
6. 文档状态、正文、分块数量更新到 MongoDB。

知识库检索：

1. MCP 工具收到 query。
2. `embedding_service.encode_query()` 生成 query embedding。
3. `qdrant_client.search_documents()` 检索 Qdrant。
4. 根据 `permission` 过滤结果。
5. 返回 context 和引用文档列表。

QA 缓存：

1. 思维链写入 MongoDB。
2. 需要缓存时，问题 embedding 写入 `QDRANT_QA_COLLECTION_NAME`。
3. 相似问题检索从 QA collection 查询。
4. 删除缓存时同步清空 `vector_id` 并删除 Qdrant point。

## 对话真实流式返回修复记录

现象：

- `/messages` 接口本身已经是 SSE，前端也使用 `fetch + response.body.getReader()` 读取流。
- 实测首个 SSE 事件很快就能到达，但最终答案会等 Agent 推理结束后一次性返回。
- 修复前一次真实请求中，首个事件约 `0.104s` 到达，首个 `answer_chunk` 约 `38.491s` 才到，而且只有 1 个大块。

原因：

- 之前能流式返回，主要依赖旧版 `internal/llm/llm_service.py::_stream_generate()` 和旧版 `MessageService._generate_ai_reply_stream()` 的队列回调机制。
- 旧版逻辑会把模型 token 通过 callback 放进队列，外层持续消费队列，一识别到 `Answer:` 就立刻发送 `answer_chunk`。
- 当前版本重构后，`internal/agent/react_agent.py` 走 LangChain `AgentExecutor.ainvoke()`，整体更像等待 Agent 完整执行结束后再拿最终 `output`。
- 同时 `internal/service/ai/stream_parser.py` 只等 `Answer:` 或 `Final Answer:`，如果模型直接输出普通答案，没有显式标记，就会一直攒到最后再由完整结果兜底发送。

修改：

- `internal/agent/react_agent.py`
  - 保留原有 `run()`，新增 `run_stream()`。
  - `run_stream()` 使用 LangChain `astream_events` 读取 `on_chat_model_stream` / `on_llm_stream`。
  - 模型 token 通过 callback 发送 `llm_chunk`。
  - 工具开始、工具结束仍发送 `action`、`observation`、`tool_result`。
  - 保留最终完整结果作为兜底，避免事件流异常时没有回答。

- `internal/service/ai/ai_reply_service.py`
  - 创建 Agent 后优先调用 `agent.run_stream()`。
  - 继续使用现有 `event_queue + StreamParser` 链路，把 `llm_chunk` 转成 `answer_chunk`。
  - 保留文档收集、RAG 结果、思维链保存、AI 消息保存等现有流程。

- `internal/service/ai/stream_parser.py`
  - 修复 `Final Answer:` 被拆成多个 chunk 时无法正确识别的问题。
  - 进入最终答案阶段后，后续 chunk 直接转成 `answer_chunk`。
  - 如果开头不是 `Thought:`、`Action:`、`Observation:`、`Answer:`、`Final Answer:` 的可能前缀，按普通答案直接进入流式输出。
  - 过滤推理标记，避免普通聊天气泡混入思考过程、工具调用过程。

- `tests/test_stream_parser.py`
  - 新增解析器回归测试，覆盖完整 `Final Answer:`、跨 chunk 的 `Final Answer:`、答案阶段连续 chunk、普通答案无标记流式输出、推理标记不混入最终答案。

## Kafka metadata 报错排障记录

本次出现过：

```txt
kafka.errors.NodeNotReadyError: NodeNotReadyError: metadata
WARNING:kafka.cluster:No node available for metadata refresh - backoff/retry
```

这不等于远程 Kafka 已经宕机。实际排查时服务器 Kafka 是正常的：

- `systemctl is-active kafka` 返回 `active`。
- `ss -lntp | grep 9092` 能看到 Kafka 监听 `0.0.0.0:9092`。
- `nc -vz 8.140.245.242 9092` 本地能连通。
- `/opt/kafka/bin/kafka-broker-api-versions.sh --bootstrap-server 127.0.0.1:9092` 能返回 `Metadata(3)` 等 API。
- `/opt/kafka/bin/kafka-topics.sh --bootstrap-server 127.0.0.1:9092 --list` 能看到业务 topic。

真正的问题在客户端侧：

- `kafka-python` 连接 Kafka `4.3.0` 时，如果不固定 `api_version`，启动阶段会自动探测 broker API。
- 自动探测期间可能出现 metadata 刷新时机不稳定，日志里会看到 `NodeNotReadyError: metadata`。
- 端口能通只能证明 TCP 通，不代表客户端 metadata 协商稳定。
- 因此本项目固定 `api_version = [4, 3]`，让 producer 和 consumer 都跳过自动探测。

本次还踩过一个配置边界：

- `request_timeout_ms` 是 producer 和 consumer 都支持的公共配置，可以放在 `[kafka]`。
- `max_block_ms` 只支持 producer，不能从公共配置传给 consumer。
- 如果把 `max_block_ms` 传给 consumer，会报 `KafkaConfigurationError: Unrecognized configs: {'max_block_ms'}`。
- 所以 `max_block_ms` 必须只放在 `[kafka.producer]`。

以后遇到 Kafka 报错按这个顺序查：

1. 先确认服务器没有宕：`ssh root@8.140.245.242 'systemctl is-active kafka && ss -lntp | grep 9092'`。
2. 再确认本地网络：`nc -vz 8.140.245.242 9092`。
3. 再确认 broker metadata：`ssh root@8.140.245.242 'export JAVA_HOME=/opt/temurin-jre-17; export PATH=/opt/temurin-jre-17/bin:$PATH; /opt/kafka/bin/kafka-broker-api-versions.sh --bootstrap-server 127.0.0.1:9092 | head'`。

## Git 推送报错排障记录

本次推送 GitHub 时出现过几类错误：

```txt
Could not resolve host: github.com
Error in the HTTP2 framing layer
Failed to connect to github.com port 443
```

处理结论：

- `Could not resolve host: github.com` 是本机 DNS 解析失败，不是 Git 仓库问题。
- 当时 `/etc/hosts` 没有被成功写入，不能假设 hosts 已经修复。
- 本机 Wi-Fi DNS 从单一 `114.114.114.114` 改成 `1.1.1.1`、`8.8.8.8`、`223.5.5.5` 后，`github.com` 可以重新解析。
- `Error in the HTTP2 framing layer` 是 HTTPS/HTTP2 传输层问题，代码和提交本身没坏。
- `Failed to connect to github.com port 443` 可能是当前解析到的 GitHub IP 不通，需要换一个可连的 GitHub IP。

以后 Git 推送失败按这个顺序查：

1. 先看 Git 状态：`git status --short --branch`。
2. 查 DNS：`networksetup -getdnsservers Wi-Fi`。
3. 查 GitHub 解析：`dscacheutil -q host -a name github.com`。
4. 查 443 端口：`nc -vz github.com 443`。
5. 如果 DNS 不稳定，设置 Wi-Fi DNS：`networksetup -setdnsservers Wi-Fi 1.1.1.1 8.8.8.8 223.5.5.5`。
6. 刷新 DNS 缓存：`dscacheutil -flushcache && killall -HUP mDNSResponder`。
7. 如果 HTTP/2 报错，使用 HTTP/1.1 推送：`git -c http.version=HTTP/1.1 push origin main`。
8. 如果 `github.com` 当前 IP 端口不通，先查可用 IP：`dig +short @1.1.1.1 github.com A`。
9. 找到可连 IP 后临时指定本次 Git 请求，不要长期写死 hosts：`git -c http.version=HTTP/1.1 -c http.curloptResolve=github.com:443:20.205.243.166 push origin main`。

本次可用经验：

- `20.200.245.247` 当时能解析出来，但连接 `github.com:443` 超时。
- `20.205.243.166` 当时可以连通 `443`，用 `http.curloptResolve=github.com:443:20.205.243.166` 后推送成功。
- 推送成功后要跑 `git fetch origin main && git status --short --branch`，确认本地 `main` 和 `origin/main` 已同步。
- 不要因为第一次 push 失败就重复提交；先确认本地提交是否已经存在，再只重试 push。

## 2026-06-24 文档上传去重与语义完整性

### 现象

- 相同文件重复上传会生成新的 Mongo 文档和新的 Qdrant 向量。
- 单个文档内部存在完全相同的文本块时，原流程会重复 embedding 并写入向量库。
- 只在检索阶段去重可以减少返回重复，但不能减少入库存储和处理耗时。

### 原因

- 上传入口原来只按 UUID 生成新文件名，没有计算文件内容哈希。
- 文档处理链路原来是全文提取后直接分块、embedding、写 Qdrant，没有入库前的精确内容去重。
- `document_uuid + chunk_index` 只能保证同一文档同一序号重复处理时覆盖，不能阻止同一文件重复上传，也不能阻止同一文档内部不同 chunk 内容相同。

### 修改

- 上传前计算文件内容 `sha256`。
- 新文档在 `extra_data` 中保存 `file_sha256`、`original_filename`、`dedup_status`。
- 上传时按 `file_sha256 + permission` 查找处理中或处理完成的已有文档；命中时直接返回已有文档，不保存新文件、不创建新 Mongo 记录、不提交处理任务。
- 文档处理时在 embedding 前做精确 chunk 去重，只合并归一化空白后完全相同的 chunk，不做相似去重。
- 处理结果写回 `raw_chunks_count`、`deduped_chunks_count`、`duplicate_chunks_removed`、`vectors_count` 和 `chunks_count`。
- Mongo `documents.content` 仍保存完整原文，去重只影响 Qdrant 入库。

## 2026-06-24 协作专家透明过程原文展示修复

- 聊天页主助手能展示思考、操作、观测过程。
- 协作专家虽然有具体专家分组，但过程内容会被压成“查询公开资料”“已获得专家处理结果”等固定话术。

## 2026-06-26 RAG 评估规则分 Sigmoid 校准映射

### 现象

- RAG 评估历史记录里出现过负的综合分，例如 `-4`、`-5` 一类分数。
- 这些负分来自重排模型的原始 `rerank_score`，不是用户可理解的 `0-1` 质量分。
- 如果简单把所有负数改成 `0`，虽然页面不再显示负分，但会丢掉不同负分之间的相关性差异。

### 原因

- BGE Reranker 这类重排模型输出的是原始相关性 logit，通常可能小于 `0`，也可能大于 `1`。
- `rerank_score` 只能表示“检索片段和问题的相关性倾向”，不能直接当作综合回答质量。
- 综合评分需要保持 `0-1`，并且要让负的重排分仍然有可比较的区分度。

### 修改

- RAG 规则评分不再把负 `rerank_score` 直接映射为 `0`。
- 现在对 reranker 原始分使用 Sigmoid 校准：

```txt
rule_score = sigmoid((rerank_score - 0.0) / 2.0)
overall_score = llm_score * 0.8 + rule_score * 0.2
```

- 如果没有 `rerank_score`，规则评分继续使用 `vector_score`，并限制在 `0-1`。
- `score_breakdown` 会保存规则分来源和映射信息：
  - `rule_score_source`
  - `rule_score_raw`
  - `rule_score_mapping`
  - `rule_score_center`
  - `rule_score_scale`
- 默认最低入队规则分调整为 `0.05`，极低相关片段不会无限进入评估队列。
- 历史记录兼容：
  - 已有 `rule_score` 的旧记录继续按旧值展示。
  - 只有旧记录缺少 `rule_score` 时，才根据原始 `rerank_score` 或 `vector_score` 动态补算。

### 口径说明

- `rerank_score` 是原始重排分，允许为负，保留在详情里用于判断检索原始质量。
- `rule_score` 是校准后的规则评分，范围固定为 `0-1`。
- `llm_score` 仍然是回答质量主评分。
- `overall_score` 是最终综合评分，范围固定为 `0-1`，`1` 为最高。
- 负的 `rerank_score` 不代表系统异常，只代表检索片段相关性较低。
