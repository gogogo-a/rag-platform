# AI 交接文档：Qdrant 向量库改造

## 接手摘要

这个项目已经从 Milvus 全面切到 Qdrant。后续 AI 或开发者接手时，不要再恢复 Milvus，也不要再引入 `pymilvus`、`langchain-milvus` 之类依赖。

接手时按以下结论执行：

- 向量库统一使用 Qdrant。
- 文档向量写入、检索、删除走 Qdrant；列表和详情的 `chunk_count` 优先读 MongoDB 已保存的 `extra_data.chunks_count`。
- QA 相似问答缓存也走 Qdrant。
- MCP 知识库搜索工具只调用 Qdrant。
- MCP 自定义工具默认全部启用，不再受 `ENABLE_EXTRA_MCP_TOOLS` 限制。
- 后续字段统一使用 `vector_id`，旧的 `milvus_id` 只作为兼容读取，不应继续新增。
- 启动时只需要 MongoDB、Redis、Qdrant、MCP 等服务，不再需要 Milvus。
- 当前 Qdrant 部署在服务器上，MongoDB 也部署在服务器上；本地没有对应服务时，不代表服务器环境不可用。
- 旧 Milvus 数据不做自动迁移，如果需要旧数据，应走单独快照导入 Qdrant。
- 页面和接口文案不要出现 Milvus，统一写“向量库”或 “Qdrant”。
- 不要恢复 Milvus 文件、Docker 配置或工具限制开关。
- 文档详情正文以 MongoDB `documents.content` 为准，Qdrant 只负责检索和向量数据。

## 当前目标

本项目已从 Milvus 向量库切换为 Qdrant。后续接手时，默认不要再恢复 Milvus，也不要再引入 `pymilvus` 或 `langchain-milvus`。

当前向量相关目标：

- 文档向量写入、检索、删除走 Qdrant。
- 文档列表和详情不要实时逐条扫描 Qdrant 统计分块数。
- QA 缓存的向量写入、相似问题检索和删除全部走 Qdrant。
- MCP 自定义工具默认全部启用。
- 用户可见页面和接口文案不出现 Milvus。

## 关键文件

- `internal/db/qdrant.py`
  - Qdrant 统一客户端封装。
  - 提供 collection 初始化、文档向量 upsert、QA 缓存 upsert、检索、按文档删除、计数、scroll 等方法。

- `main.py`
  - 启动时会初始化 Qdrant 文档集合和 QA 缓存集合。
  - 启动流程不再连接 Milvus。

- `internal/document_client/document_processor.py`
  - 文档解析后生成 embedding，并写入 Qdrant。
  - 处理完成后把完整解析正文回写到 MongoDB `documents.content`。
  - 删除任务按 `document_uuid` 删除 Qdrant 向量。

- `internal/service/orm/document_sever.py`
  - 文档详情和列表的 `chunk_count` 优先读取 MongoDB 里保存的分块数量。
  - 上传接口只保存文件、创建记录、提交后台任务，不在请求内同步解析全文。
  - 删除文档时同步删除 Qdrant 向量。

- `pkg/agent_tools_mcp/knowledge_search_mcp.py`
  - MCP 知识库检索直接调用 Qdrant。
  - 普通用户仍过滤 `permission=1` 的文档。

- `internal/service/ai/thought_chain_store.py`
  - QA 缓存写入 Qdrant。
  - 使用 `vector_id` 记录向量 ID。

- `internal/service/ai/similar_qa_cache.py`
  - 相似问答检索和缓存删除走 Qdrant。

- `internal/service/orm/qa_cache_service.py`
  - QA 缓存管理以 MongoDB 的 `ThoughtChainModel` 为主，Qdrant 只负责向量检索和删除。

- `internal/service/visualization/document_3d_service.py`
  - 3D 可视化从 Qdrant scroll 读取向量。

- `pkg/agent_tools_mcp/mcp_config.py`
  - MCP 工具默认全部加载，不再使用 `ENABLE_EXTRA_MCP_TOOLS`。

## 配置项

当前部署环境：

- Qdrant 在服务器上。
- MongoDB 在服务器上。
- 后端应通过环境变量连接服务器上的 Qdrant 和 MongoDB。
- 本地开发机如果没有启动 Qdrant 或 MongoDB，不应据此判断服务器连接异常。

`.env` / `env_template.txt` 中向量库配置为：

```env

QDRANT_PORT=6333
QDRANT_COLLECTION_NAME=documents
QDRANT_QA_COLLECTION_NAME=qa_cache
VECTOR_DIMENSION=1024
```

依赖中应保留：

```txt

```

依赖中不应出现：

```txt

langchain-milvus
```

## 数据流

文档上传：

1. API 保存文件和 MongoDB 文档记录。
2. API 提交后台处理任务并立即返回处理中。
3. `document_processor` 解析文本并分块。
4. `embedding_service` 生成向量。
5. `qdrant_client.upsert_documents()` 写入 Qdrant。
6. 文档状态、正文、分块数量更新到 MongoDB。

知识库检索：

1. MCP 工具收到 query。
2. `embedding_service.encode_query()` 生成 query embedding。
3. `qdrant_client.search_documents()` 检索 Qdrant。
4. 根据 `permission` 过滤结果。
5. 返回 context 和引用文档列表。

QA 缓存：

1. 思维链写入 MongoDB。
2. 需要缓存时，问题 embedding 写入 `QDRANT_QA_COLLECTION_NAME`。
3. 相似问题检索从 QA collection 查询。
4. 删除缓存时同步清空 `vector_id` 并删除 Qdrant point。

## 文档管理性能与详情内容规则

- 上传接口不要同步解析全文。大文件解析和向量化统一交给后台处理任务。
- 后台处理成功后必须把解析后的完整正文写回 `documents.content`，详情页正文只读这个字段。
- 列表和详情的分块数量优先取 `extra_data.chunks_count`，其次取 `page`，不要为每条文档实时扫描远程 Qdrant。
- Qdrant payload 可用于检索和 3D 可视化，不作为文档详情正文的数据源。
- 历史文档如果 `content` 为空，使用 `python scripts/backfill_document_content.py` 从 `uploads/` 原文件补齐正文。

## Kafka 远程部署与连接规则

- Kafka 统一使用服务器 `8.140.245.242:9092`，本地项目不要再连接 `127.0.0.1:9092` 或 `localhost:9092`。
- 本地没有 Kafka 不代表异常；文档处理队列以服务器 Kafka 为准。
- 服务器 Kafka 使用原生安装，不使用 Docker，不依赖 ZooKeeper。
- 服务器部署为 Kafka `kafka_2.13-4.3.0` KRaft 单节点模式。
- Java 使用 `/opt/temurin-jre-17`，服务文件里显式设置 `JAVA_HOME=/opt/temurin-jre-17`。
- Kafka 安装目录为 `/opt/kafka_2.13-4.3.0`，软链为 `/opt/kafka`。
- Kafka 数据目录为 `/var/lib/kafka/kraft-combined-logs`。
- Kafka 配置文件为 `/opt/kafka/config/kraft/server.properties`。
- `listeners` 为 `PLAINTEXT://0.0.0.0:9092,CONTROLLER://127.0.0.1:9093`。
- `advertised.listeners` 为 `PLAINTEXT://8.140.245.242:9092`。
- `controller.quorum.voters` 为 `1@127.0.0.1:9093`。
- Kafka 已注册为 systemd 服务 `kafka.service`，并设置开机自启。
- Kafka 服务设置 `KAFKA_HEAP_OPTS=-Xms512m -Xmx512m`，适配当前小内存服务器。
- 当前已创建 topic：`document_embedding`、`document_delete`、`batch_processing`。
- 本地 Docker 容器 `brainwave_kafka_ui`、`brainwave_kafka`、`brainwave_zookeeper` 应保持停止，并关闭自动重启。
- 如果本地连接 `8.140.245.242:9092` 失败，先检查阿里云安全组是否放行 TCP `9092`，建议只允许当前本机公网 IP 访问。
- 之前 Kafka 报错的主要原因是本地配置和客户端回退地址仍可能走本地 `localhost` / `127.0.0.1`，且本地服务状态不应作为现在的判断依据。
- 服务器直接从 Apache 下载 Kafka 如果很慢，可以使用服务器可访问的镜像下载；也可以本地下载后通过 `scp` 上传到服务器。
- 服务器安装包可用命令：`wget https://mirrors.tuna.tsinghua.edu.cn/apache/kafka/4.3.0/kafka_2.13-4.3.0.tgz -O /tmp/kafka_2.13-4.3.0.tgz`。
- 本地上传备用命令：`scp /tmp/kafka_2.13-4.3.0.tgz root@8.140.245.242:/tmp/`。
- 验证 Kafka 服务：`ssh root@8.140.245.242 'systemctl is-active kafka && ss -lntp | grep 9092'`。
- 验证本地连通：`nc -vz 8.140.245.242 9092`。
